Where to go from here?

The application we started to build today is obviously not finished :) If you wish to do so for your own learning and fun, consider the following:

Send emails to donors when a new donation event is created in their city

Even though the intro pages mentioned this feature there was no time to actually make it. We can however briefly talk about how we could go about making it.

As with every answer in tech, it depends, and there are more than one way of doing it. Below are two event-driven approaches I would consider.

DynamoDB streams

It is possible to configure your DynamoDB table to emit events when item-level changes occur - this is called a “stream”. They are used to enable post-processing on the data that was written to the table. When enabled, you can configure a stream and choose the information that will be written to the stream whenever the data in the table is modified:

  • KEYS_ONLY — Only the key attributes of the modified item.
  • NEW_IMAGE — The entire item, as it appears after it was modified.
  • OLD_IMAGE — The entire item, as it appeared before it was modified.
  • NEW_AND_OLD_IMAGES — Both the new and the old images of the item.

Another great feature of DynamoDB streams is that they can be filtered by the content in the item; for example, we can create a stream with a filter that would only emit an event if a new donation event was created, not when a new donor is created. This saves costs of the whole application because downstream services would only receive relevant items.

As a target of the event we can use a Lambda function. It would receive the event and through templating or string manipulation create an email message. It would not send the email though - there’s another service for that. A picture says a thousand words:

The left part of the diagram was omitted since we saw it and built it already. What the architecture above shows is an addition of:

  • another Lambda function, which:
    • receives the stream event with a city where the new donation event will take place
    • executes a query on the DynamoDB table to fetch all donors living in that city
    • generates the email message (HTML)
    • invokes SES (Simple Email Service)
  • SES, which will do the heavy lifting of sending those emails that the Lambda function generated

Drawbacks:

  • SES can send an email to up to 50 recipients at a time, so a “paging” logic must be included (Step Functions would shine here)

Notifications

A different, although similar, approach could be to publish a message from the donation_create() function. Whenever a donation is created, we can publish a message to a Simple Notification Service (SNS) topic. As the name suggests, SNS is a notification service; it works with topics to which subscribers subscribe to. Whenever there is a new message in the topic, SNS will make sure to notify all those subscribers with one of:

  • email
  • email-JSON
  • SMS
  • http(s)
  • mobile push notifications
  • Lambda function
  • others

In our case, whenever a donor registers we could subscribe them to a topic. They would receive an email with a confirmation link, and after confirming they would become a confirmed subscriber.

After that, whenever a new donation event is created we publish a message to that topic and SNS will take care of sending a notification to every confirmed subscriber.

Drawbacks:

  • as the name suggests, it is a simple notification service, meaning it sends only plaintext emails; you can include links though
